# AGENTS.md

## プロジェクト概要

このプロジェクトは、Dynalist のような Web ベースのアウトラインエディタです。

目的は、デスクトップでもモバイルでも快適に使える、軽量なツリーエディタを作ることです。サーバ側には PHP を使います。クライアント側は、特別な理由がない限り、素の JavaScript で実装します。

この `AGENTS.md` は、ユーザーから追加方針が出るたびに逐次更新します。新しい指示が既存方針と矛盾する場合は、新しい指示を優先してこのファイルを書き換えます。

重視する点:

- すばやく編集できること
- ツリー構造が分かりやすいこと
- キーボード操作が快適なこと
- タッチ操作が快適なこと
- 配布と設置が簡単なこと
- コードが読みやすいこと

## 技術方針

基本構成は次の通りです。

- サーバ: PHP
- データベース: SQLite
- フロントエンド: 素の JavaScript、HTML、CSS
- 通信: `fetch()` による JSON 通信
- テンプレート: `templates/` 以下の PHP/HTML 断片を素の PHP で合成する

明示的に依頼されない限り、React、Vue、Svelte、TypeScript、ビルドツール、フロントエンド用パッケージマネージャは導入しません。

PHP フレームワークやテンプレートエンジンも導入しません。`require`、関数、小さなヘルパーを使い、素の PHP で読みやすく構成します。

Rust は、将来的にツリー変換、検索、インポート/エクスポート、WebAssembly 実験などの独立したロジックに使う余地があります。ただし、最初の実装ではブラウザ UI は素の JavaScript で作ります。

## 推奨ディレクトリ構成

シンプルな構成を優先します。

```text
/
  index.php
  router.php
  api/
    lib/
      auth.php
      db.php
      template.php
    tree.php
    node_create.php
    node_update.php
    node_delete.php
    node_move.php
  templates/
    layout.php
    login.php
    outline.php
  assets/
    app.js
    style.css
  data/
    .htaccess
    users.sqlite
    user/
      <user_id>/
        index.sqlite
        upload/
```

ファイル名は必要に応じて変更して構いません。ただし、役割の境界は明確にします。

- `index.php`: ルーティングや初期表示のための簡単な呼び出しだけを行う
- `router.php`: PHP内蔵サーバで開発確認するときに、`data/` の直接配信を防ぐ
- `api/lib/*.php`: 認証、DB 接続、テンプレート合成などの共通処理を担当する
- `templates/*.php`: HTML を保持する。PHP の処理ロジックを詰め込みすぎない
- `assets/app.js`: クライアント側の状態管理、描画、UI操作を担当する
- `assets/style.css`: レイアウトと見た目を担当する
- `api/*.php`: 小さな JSON API を提供する
- `data/users.sqlite`: ユーザー一覧、ログイン情報、ユーザー用ディレクトリ識別子を保存する
- `data/user/<user_id>/index.sqlite`: 1ユーザーにつき1つのアウトライン用 SQLite ファイル
- `data/user/<user_id>/upload/`: 将来、素材ファイルをアップロードして使うための保存先
- `data/.htaccess`: Apache で設置した場合に、DBやアップロード素材の直接配信を防ぐ

## データモデル

ユーザーログインは必須です。ログインしていないユーザーはアウトライン API を利用できません。

データベースは2種類に分けます。

- `data/users.sqlite`: 全体のユーザー管理
- `data/user/<user_id>/index.sqlite`: 各ユーザー専用のアウトラインデータ

`<user_id>` はディレクトリ名として安全な値にします。メールアドレスや表示名をそのまま使わず、内部IDまたはランダムな識別子を使います。

ユーザー管理DBの推奨テーブル:

```sql
CREATE TABLE users (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  user_dir TEXT NOT NULL UNIQUE,
  login_id TEXT NOT NULL UNIQUE,
  password_hash TEXT NOT NULL,
  display_name TEXT NOT NULL DEFAULT '',
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);
```

パスワードは必ず `password_hash()` で保存し、検証には `password_verify()` を使います。平文パスワードを保存してはいけません。

アウトラインDBでは、ノードを隣接リスト方式で管理します。

ユーザー別 `index.sqlite` の推奨テーブル:

```sql
CREATE TABLE nodes (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  parent_id INTEGER NULL,
  position INTEGER NOT NULL DEFAULT 0,
  text TEXT NOT NULL DEFAULT '',
  collapsed INTEGER NOT NULL DEFAULT 0,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  FOREIGN KEY (parent_id) REFERENCES nodes(id) ON DELETE CASCADE
);
```

各項目の意味:

- `id`: ノードID
- `parent_id`: 親ノードID。ルート直下なら `NULL`
- `position`: 同じ親を持つ兄弟ノード内での並び順
- `text`: ノード本文
- `collapsed`: 折りたたみ状態。`0` または `1`
- `created_at`: 作成日時
- `updated_at`: 更新日時

ノード本文とツリー構造は分けて扱います。DOM の並び順をデータ本体として扱ってはいけません。

将来の素材アップロードに備え、ファイル本体は `data/user/<user_id>/upload/` に保存します。ファイルのメタデータを管理する場合は、ユーザー別 `index.sqlite` に `uploads` テーブルを追加します。

## クライアント側の状態管理

JavaScript 側では、アウトラインの状態を構造化して保持します。

例:

```js
const state = {
  nodes: new Map(),
  rootIds: [],
  activeNodeId: null,
  dirty: false
};
```

アプリケーションの状態が正です。DOM は、その状態を表示した結果にすぎません。

`<li>` などの DOM 要素を走査して、毎回ツリー構造を復元する設計は避けます。先に状態を更新し、その結果として再描画します。

## 描画方針

アウトラインは状態から描画します。

各ノード要素には、少なくとも次の情報を持たせます。

- `data-id` によるノードID
- 深さに応じたインデント
- 子ノードがある場合の折りたたみボタン
- 編集可能な本文欄。`contenteditable` または軽量な入力要素を使う

最初のバージョンでは、構造変更後に全体を再描画しても構いません。性能が実際に問題になってから最適化します。

## 基本編集操作

まず次の操作を実装します。

- ノードをクリックまたはタップすると本文を編集できる
- `Enter`: 現在のノードの下に兄弟ノードを追加する
- `Tab`: 現在のノードを1段深くする
- `Shift+Tab`: 現在のノードを1段浅くする
- 空のノードで `Backspace`: ノードを削除する、または前のノードと結合する
- 矢印キー: 表示中のノード間を移動する
- 折りたたみボタン: 子ノードを折りたたむ、または展開する

ブラウザ標準の挙動がエディタ操作を壊さないように、重要なキーは明示的に処理します。

## タッチ操作

タッチデバイスでも快適に使えるようにします。

推奨ジェスチャー:

| 操作 | 動作 |
|---|---|
| ノード本文をタップ | 本文を編集 |
| 右へスワイプ | ノードを1段深くする |
| 左へスワイプ | ノードを1段浅くする |
| 長押し | 移動モード、またはドラッグモードを開始 |
| 折りたたみアイコンをタップ | 折りたたみ、または展開 |

マウス操作とタッチ操作を別々に実装するのではなく、Pointer Events を使います。

使用するイベント:

- `pointerdown`
- `pointermove`
- `pointerup`
- `pointercancel`

スワイプ判定:

- `pointerdown` で `startX` と `startY` を記録する
- `pointerup` で `dx` と `dy` を計算する
- `Math.abs(dx)` が十分に大きく、かつ `Math.abs(dy)` より明らかに大きい場合だけ、左右スワイプとして扱う
- しきい値は、まず `50px` 前後を目安にする

推奨 CSS:

```css
.node {
  touch-action: pan-y;
  user-select: none;
}

.node-text {
  user-select: text;
}
```

可能なら、スワイプ中に軽いプレビューを表示します。

- 右スワイプ中: `子にする`
- 左スワイプ中: `親に戻す`

## インデントとアウトデントの規則

インデントの規則:

- ノードを1段深くすると、直前の表示中の兄弟ノードの子になります。
- 直前の兄弟ノードがない場合、インデントはできません。
- インデント後は、新しい親ノードの子リストの末尾に配置します。

例:

```text
- 第1章
- 第2章
```

`第2章` をインデントすると、次のようになります。

```text
- 第1章
  - 第2章
```

アウトデントの規則:

- ノードを1段浅くすると、現在の親ノードと同じ階層へ移動します。
- 移動後は、元の親ノードの直後の兄弟になります。
- ルート直下のノードは、それ以上アウトデントできません。

ノードを移動するときは、その子孫ノードも一緒に移動します。

上下移動の規則:

- ノードを1つ上に移動すると、同じ階層の直前の兄弟ノードと位置を入れ替えます。
- 直前の兄弟ノードがない（その親の最初のノードである）場合、親と同じ階層にアウトデントし、親の直前に配置されます。
- ルート直下の最初のノードは、それ以上上に移動できません。
- ノードを1つ下に移動すると、同じ階層の直後の兄弟ノードと位置を入れ替えます。
- 直後の兄弟ノードがない（その親の最後のノードである）場合、親と同じ階層にアウトデントし、親の直後に配置されます。
- ルート直下の最後のノードは、それ以上下に移動できません。

## 保存方針

API との通信は `fetch()` と JSON で行います。

本文の更新では debounce を使い、キー入力のたびに保存リクエストを送らないようにします。

推奨:

- 本文編集: 300 から 800 ミリ秒程度の debounce 保存
- 構造変更: 状態更新後にすぐ保存
- 折りたたみ状態: 保存する。ただし本文や構造より優先度は低い

必要なら、小さな保存状態を表示します。

- 保存中
- 保存済み
- 保存失敗

保存に失敗しても、ローカルの編集内容は保持し、ユーザーが編集を続けられるようにします。

## API 設計

API は小さく、分かりやすく保ちます。

推奨エンドポイント:

```text
GET  /api/tree.php
POST /api/node_create.php
POST /api/node_update.php
POST /api/node_delete.php
POST /api/node_move.php
```

すべての API は JSON を返します。

成功レスポンス例:

```json
{
  "ok": true,
  "node": {
    "id": 1,
    "parent_id": null,
    "position": 0,
    "text": "Example",
    "collapsed": 0
  }
}
```

エラーレスポンス例:

```json
{
  "ok": false,
  "error": "Node not found"
}
```

可能な範囲で適切な HTTP ステータスコードを使います。ただし、クライアント側の処理は複雑にしすぎません。

## PHP 実装方針

PHP は素の PHP で書きます。

SQLite へのアクセスには PDO を使います。

`index.php` は薄く保ちます。認証確認、テンプレート選択、最低限の初期化だけを行い、HTML のまとまった構造は `templates/` に置きます。

例:

```php
require __DIR__ . '/api/lib/auth.php';
require __DIR__ . '/api/lib/template.php';

$user = current_user();
echo render_template($user ? 'outline.php' : 'login.php', ['user' => $user]);
```

この例は方針を示すものです。実装時は必要に応じて `layout.php` などで共通レイアウトを合成します。

API ファイルでは JSON ヘッダーを設定します。

```php
header('Content-Type: application/json; charset=utf-8');
```

JSON リクエストボディは次のように読み取ります。

```php
$input = json_decode(file_get_contents('php://input'), true);
```

入力値は必ず検証します。

- ログイン済みユーザーであること
- ID は整数であること
- 本文は文字列であること
- `parent_id` は `null` か、存在するノードIDであること
- ツリー内に循環を作らないこと
- クライアントから渡された `position` を無条件に信用しないこと
- 対象DBが現在のログインユーザーの `data/user/<user_id>/index.sqlite` であること

移動、インデント、アウトデント、削除など、構造を変更する処理ではトランザクションを使います。

ユーザー別DBファイルは、ログインユーザーから解決します。API リクエストで任意のDBパスや任意の `user_id` を指定できる設計にしてはいけません。

## 認証方針

ユーザーログインは最初のバージョンから必須です。

最低限必要な機能:

- ログイン画面を表示する
- `login_id` とパスワードで認証する
- PHP セッションでログイン状態を保持する
- ログアウトできる
- ログインしていない場合、アウトライン画面と書き込み API を使わせない

ユーザー追加方法は最初は簡素で構いません。例えば、管理用スクリプトや初期セットアップ処理でユーザーを作成してもよいです。ただし、アウトライン編集画面は必ずログイン後に表示します。

セッション開始時は `session_start()` を使います。ログイン成功時には `session_regenerate_id(true)` を呼びます。

## セキュリティ

PHP から HTML を出力するときは、ユーザー本文を必ずエスケープします。

JavaScript では、ユーザー本文の表示に `textContent` を優先します。ユーザー本文を `innerHTML` に直接代入してはいけません。

書き込み API はログイン必須です。認証なしで公開して安全だと考えてはいけません。

ログインフォームと書き込み API には CSRF 対策を検討します。最初の実装でも、可能ならセッションに紐づく CSRF トークンを使います。

ユーザー別のデータディレクトリを扱うときは、パストラバーサルを防ぎます。`../` や任意パスを受け取らず、`users.sqlite` に保存された `user_dir` からのみパスを組み立てます。

`data/` は Web ルート配下にあるため、DBファイルやアップロード素材を直接配信しない対策を必ず入れます。Apache では `data/.htaccess` に `Require all denied` を置きます。SQLiteファイルとアップロードされた素材ファイルは生成物として扱い、通常はGit管理に含めません。

素材ファイルのアップロード機能を追加するときは、少なくとも次を検証します。

- ファイルサイズ
- MIME タイプ
- 拡張子
- 保存ファイル名
- ログインユーザーの保存先であること

アップロードされたファイルをそのまま PHP として実行できる場所に置いてはいけません。

## スタイル方針

デザインは実用的で、編集作業を中心にします。

優先すること:

- コンパクトな行
- 読みやすい文字サイズ
- 分かりやすいインデント
- 現在のノードが分かる表示
- モバイルで押しやすい操作領域
- 過剰な装飾を避ける

重い UI フレームワーク、大きな CSS リセット、編集速度を落とす装飾は避けます。

## アクセシビリティ

可能な範囲で意味のある HTML を使います。

推奨:

- 折りたたみ操作には `button` を使う
- フォーカス状態を見えるようにする
- キーボード操作に対応する
- 十分なコントラストを確保する
- ポインタ操作だけに依存しない

タッチジェスチャーは便利機能として用意しますが、キーボード操作や画面上の操作ボタンでも同じことができるようにします。

## インポートとエクスポート

早い段階で Markdown エクスポートを実装すると便利です。

例:

```text
- 本
  - 第1章
  - 第2章
```

Markdown インポートは後から追加します。インデントとリスト記号を解析してツリーに変換します。

最初のバージョンでインポート/エクスポートを作り込みすぎないようにします。

## 実装の優先順位

次の順番で作ります。

1. `templates/` と薄い `index.php` の構成を作る
2. `data/users.sqlite` とログイン処理を作る
3. ログインユーザーごとの `data/user/<user_id>/index.sqlite` を作成、接続する
4. ノードをツリーとして表示する
5. ノードを作成、編集できるようにする
6. PHP + SQLite で保存と読み込みを行う
7. キーボード操作を追加する
8. 折りたたみと展開を追加する
9. タッチ操作による左右スワイプのインデント/アウトデントを追加する
10. ドラッグ、または長押し移動モードを追加する
11. 検索を追加する
12. Markdown エクスポートを追加する
13. 素材ファイルのアップロード機能を追加する

## 最初のバージョンでやらないこと

明示的に依頼されない限り、最初のバージョンでは次の機能を実装しません。

- リアルタイム共同編集
- 複雑な権限管理
- オフライン同期
- CRDT
- 本格的な全文検索エンジン
- フロントエンドフレームワークへの移行
- Rust/WASM コア
- モバイルアプリ化
- 複数ユーザー間での共有機能
- 素材ファイルの高度な管理機能

これらは将来的には有用ですが、最初のバージョンでは編集体験を確認することを優先します。

## コーディングスタイル

読みやすく、変更しやすいコードを優先します。

分かりやすい名前の小さな関数を使います。

- `renderTree()`
- `createNodeAfter(id)`
- `indentNode(id)`
- `outdentNode(id)`
- `moveNode(id, newParentId, newPosition)`
- `saveNodeText(id, text)`
- `loadTree()`

コメントは、ツリー移動の規則など、読んだだけでは分かりにくい処理に絞って書きます。

パターンが見える前に、凝った抽象化を作らないでください。

## 重要な原則

このエディタは、速く、直接的で、落ち着いて使えることを目指します。

最初の成功条件は、機能の多さではありません。ノードの追加、編集、インデント、アウトデント、保存が、信頼できる手触りで動くことです。
